# Company-OS Skill

Company-OS renders a Terraria-style 2D virtual office in the browser.
It scans a Node.js project, produces a `context.json` snapshot, and populates
the canvas with agent avatars representing team members.

**The server makes zero AI API calls.** It is a purely passive visual engine.
You — Claude Code — generate all agent responses and send them to the server
via `curl` / REST. No `ANTHROPIC_API_KEY` or `.env` file is required in the
project being visualised.

---

## When to Activate This Skill

Activate when the user says anything like:

- "Start my virtual office"
- "Open the Company-OS dashboard"
- "Show me the team in the virtual office"
- "Scan this project"
- "Convene a daily meeting"
- "Ask the Backend agent what it thinks of this"
- "Make the Frontend agent speak"
- "Hold a sprint review / standup / boardroom meeting"

---

## Step 1 — Detect and Start the Server

Before issuing any agent commands, verify the server is running:

```bash
curl -s http://localhost:3000/api/health
```

If that returns a JSON response the server is already running — skip to Step 2.

If the `curl` call fails (connection refused), start the server:

```bash
npx company-os start
```

Tell the user to open `http://localhost:3000` in their browser once the line
`Company-OS is running!` appears in the terminal.

To use a different port: `npx company-os start --port 4000`

---

## Step 2 (Optional) — Scan the Project

Run a scan to populate `context.json` so agents have accurate project knowledge:

```bash
npx company-os scan
```

Or trigger it via REST without restarting:

```bash
curl -s -X POST http://localhost:3000/api/project/scan
```

---

## Step 3 — Controlling Agents (Active Orchestration)

You are responsible for generating all agent content using your own language
model and sending it to the server. The server only renders what you give it.

### A — Making an Agent Speak

When the user asks something like "What does the Frontend agent think of this?":

**1. Get the agent list** to find the agent's ID:
```bash
curl -s http://localhost:3000/api/agents
```

**2. Fetch the agent's context** (system prompt, memory, current task):
```bash
curl -s http://localhost:3000/api/agents/<id>/context
```

**3. Generate a response** internally using the returned `systemPrompt` and
`memory` as context for your own model, then post it to the server:
```bash
curl -s -X POST http://localhost:3000/api/agents/<id>/speak \
  -H "Content-Type: application/json" \
  -d '{"question": "<original question>", "answer": "<your generated response>"}'
```

Always echo the agent's response back to the user in the chat so they can
read it without needing to watch the browser.

### B — Running a Team Meeting

When the user asks to convene a meeting, execute the three steps below.
Each REST call returns immediately — the server handles all animation timing
internally, so no waiting is needed between calls.

**1. Convoke the meeting** (agents walk to the meeting room):
```bash
curl -s -X POST http://localhost:3000/api/visual/meetings/convoke \
  -H "Content-Type: application/json" \
  -d '{"type": "daily", "topic": "Daily Standup", "attendees": ["<id-1>", "<id-2>"]}'
```
Valid `type` values: `daily`, `sprint`, `boardroom`, `emergency`, `retro`, `custom`.
Omit `attendees` or pass `[]` to include all agents.

**2. For each agent**, generate what they would say given their role and the
project context, then post it:
```bash
curl -s -X POST http://localhost:3000/api/visual/meetings/speak \
  -H "Content-Type: application/json" \
  -d '{"agentId": "<id>", "message": "<generated speech>"}'
```

**3. Conclude the meeting** with a summary of outcomes:
```bash
curl -s -X POST http://localhost:3000/api/visual/meetings/conclude \
  -H "Content-Type: application/json" \
  -d '{"conclusion": "<summary>", "actionItems": ["<item 1>", "<item 2>"], "mood": "positive"}'
```
Valid `mood` values: `positive`, `neutral`, `concerned`.

## REST API Reference

All endpoints run on `http://localhost:3000` (or the configured port).

### Agents

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/api/agents` | List all agents and teams |
| `GET` | `/api/agents/:id` | Get a single agent's full data |
| `GET` | `/api/agents/:id/context` | Get `{ systemPrompt, memory, currentTask }` for prompt building |
| `POST` | `/api/agents/:id/speak` | Render speech bubble. Body: `{ question, answer }` |
| `POST` | `/api/agents` | Create agent. Body: `{ name, role, teamId, avatar?, customSystemPrompt? }` |
| `DELETE` | `/api/agents/:id` | Remove an agent |
| `POST` | `/api/agents/:id/move` | Move agent to another team. Body: `{ newTeamId }` |
| `POST` | `/api/teams` | Create a team. Body: `{ name, color }` |

### Project

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/api/project/context` | Return the current `context.json` snapshot |
| `POST` | `/api/project/scan` | Trigger a re-scan without restarting the server |

### Meetings

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/api/visual/meetings/convoke` | Start a meeting. Body: `{ type, topic, attendees? }` |
| `POST` | `/api/visual/meetings/speak` | Agent speaks in meeting. Body: `{ agentId, message }` |
| `POST` | `/api/visual/meetings/conclude` | End meeting. Body: `{ conclusion, actionItems, mood }` |
| `GET` | `/api/visual/meetings/current` | Get active meeting state and phase |
| `GET` | `/api/visual/meetings` | Meeting history (last 20 entries) |
| `POST` | `/api/visual/meetings/emergency` | Trigger emergency meeting. Body: `{ reason }` |

### Utility

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/api/health` | Health check — returns `{ status, projectName, agents, teams }` |

## Your Responsibilities as Claude Code

Company-OS is a **passive skill**. The server never initiates agent speech on
its own. Every time the user asks to debate an idea, consult an agent, or run
a meeting, you must:

1. Generate the relevant text using your own model (use the agent's `systemPrompt` and `memory` as context).
2. Send it to the server via `Bash` `curl` commands.
3. Relay what each agent said back to the user in the chat.
